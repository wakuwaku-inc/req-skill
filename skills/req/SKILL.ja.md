# /req — 開発要件ヒアリングスキル (日本語版ドキュメント)

> **注意:** これはドキュメント用の翻訳です。Claude Code が実行時にロードするのは英語版の `skills/req/SKILL.md` です。
> このファイルは `name:` フロントマターを持たず、Claude Code の自動ロード対象にはなりません。
> 英語版に変更を加えた場合は、必ずこのファイルも同期してください (詳細は `CLAUDE.md` の同期ルールを参照)。

**起動時の宣言:** 「/req を使って要件を聞き取り、`docs/requirements/YYYY-MM-DD_<title>.md` に要件定義書を作成します。」

## 入力

- `$ARGUMENTS` — ユーザーが `/req` の後に入力した `[what]` テキスト。空でも可。
- このプラグイン内の `templates/requirements.md` — テンプレートの正本。開始時に Read で読み込む。

## 絶対ルール

1. **答えを捏造しない。** プレースホルダの値がユーザーの言葉から直接得られない場合は必ず質問する。
2. **出力言語は日本語。** ユーザーとの対話は日本語で行う。内部推論は英語でも可。
3. **出力パス** は `<CWD>/docs/requirements/YYYY-MM-DD_<title>.md`。`docs/requirements/` が無ければ作成。このディレクトリ外には書き込まない。
4. **自動コミット禁止。** ファイル書き出しまで。git 操作はユーザーの判断に委ねる。

## プレースホルダ (templates/requirements.md と同期)

| プレースホルダ | 対応する質問 |
| --- | --- |
| `{{title}}` | 最後に内容から自動生成、ユーザーに確認。 |
| `{{issues}}` | 💡 なぜ必要? — 課題・困り事・背景 |
| `{{issue_urls}}` | 🔗 関連URL (Slack / Chatwork / チケット等) |
| `{{requests}}` | 🎯 何を作りたい? — 要望・手段 |
| `{{requirements}}` | 📱 要件 — 画面 / 機能 / 項目 / 動作 |
| `{{designs}}` | 🎨 デザインURL |
| `{{references}}` | 参考サイト |
| `{{others}}` | その他資料 |
| `{{due}}` | 希望納期 |
| `{{due_reasons}}` | この日までに必要な理由 |
| `{{DoD}}` | ✅ 何ができればOK? (チェックリスト) |
| `{{verify}}` | 誰がどうやって確認するか |
| `{{notes}}` | その他・補足 |

## フロー

### Step 0 — 引数ハンドリング
`$ARGUMENTS` が空なら「何について要件を作りますか？」と質問して入力を待つ。

### Step 1 — テンプレート読み込み
このプラグインの `templates/requirements.md` を Read する。無ければプラグイン再インストールを案内して中断。

### Step 2 — 初期草案作成 (ハイブリッド方式)
`$ARGUMENTS` から `{{issues}}` と `{{requests}}` の初期案を生成し、以下の形式で確認:

「以下の理解で合っていますか？修正点があれば教えてください。
- 課題・困り事・背景 (`{{issues}}`): <草案>
- 要望・手段 (`{{requests}}`): <草案>」

修正/確認を受けて草案を確定する。

### Step 3 — 順次深掘り (空欄項目のみ)
以下の順序:

1. `{{issues}}` — Step 2 の草案を確定。
2. `{{issue_urls}}` — 「課題・困り事・背景に関連するURLはありますか？ Slack / Chatwork / チケット等、複数ある場合は全て教えてください。」
3. `{{requests}}` — Step 2 の草案を確定。
4. `{{requirements}}` — 要件を深掘り。不明点があれば: 「持ち帰って調べてから戻ってきてください。続きから再開できます。」情報が揃ったら画面/機能/項目/動作の構造で `{{requirements}}` を提案し確認。
5. `{{designs}}` — 「デザインに関するURLはありますか？」
6. `{{references}}` — 「参考サイトはありますか？」
7. `{{others}}` — 「その他の資料・参考情報はありますか？」
8. `{{due}}` — 「希望納期はありますか？」
9. `{{due_reasons}}` — `{{due}}` がある場合のみ: 「なぜその日までに必要ですか？ 軽微なら『特になし』でOK。」
10. `{{DoD}}` — 要件から DoD チェックリストを推測して提案、「以下が完成条件で合っていますか？」で確認。
11. `{{verify}}` — 「完成の確認は誰がどうやって行いますか？」
12. `{{notes}}` — 「その他・補足はありますか？」

### Step 4 — タイトル生成
収集内容から 2-3 案のタイトルを提案:
「タイトル候補:
1. <候補A>
2. <候補B>
3. <候補C>
どれがよいですか？ (独自案もOK)」
選択を `{{title}}` に入れる。

### Step 5 — プレースホルダ置換とクリーンアップ
テンプレの `{{placeholder}}` を全て値に置換。ユーザーが指定しなかった (空のままの) プレースホルダは、該当行のみ削除、見出しは残す。

### Step 6 — 出力先の計算と確認
- `DATE` = 今日の日付 (JST 優先) を `YYYY-MM-DD` で。
- `TITLE` = ユーザー確定タイトル。ASCII 空白はハイフンに置換。日本語はそのまま。
- `TARGET` = `<CWD>/docs/requirements/{DATE}_{TITLE}.md`。

既存ならユーザーに: 上書き / `_2` サフィックス別名 / 中断 を確認。`_2` はインクリメント。

確定パスをユーザーに提示し、書き出し前に最終確認。

### Step 7 — ファイル書き出し
Write でファイル生成。`docs/requirements/` が無ければ作成。書き込みエラー時はエラー内容提示＋リトライ/パス変更を提案。

### Step 8 — 完了報告
絶対パスを表示。`git add` / `git commit` はしない。

## 早期終了
Step 5 前にユーザーが「ここまでで」「もういい」と言ったらそのまま Step 5 へ。未確定の `{{foo}}` はそのまま残してファイル書き出し (後から編集しやすいよう)。

## やってはいけないこと
- URL, 日付, 要件の詳細を勝手に捏造
- `<CWD>/docs/requirements/` 以外への書き込み
- 自動 git コミット / git 操作
- 実行時に `templates/requirements.md` 以外のプラグインファイルを読み書き
